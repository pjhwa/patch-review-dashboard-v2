1
Preprocessing (Python)
Python script converts raw vendor data into normalized PreprocessedPatch objects. Each product has its own preprocessingScript and preprocessedPatchMapper() function. Output is stored in the SQLite PreprocessedPatch table.
Python 3 RawPatch → PreprocessedPatch Prisma upsert
2
AI Review Loop (OpenClaw) — see "AI Review Loop" tab for full detail
Preprocessed patches are split into batches of 5 and sent sequentially to OpenClaw AI via a spawned child process (openclaw agent --agent main --json --timeout 1800). AI assigns Criticality (Critical / Important / Moderate / Low) and Decision (Done / Exclude / Pending) per patch, and writes a Korean description. Output JSON is validated with Zod; on failure the exact error is injected back into the prompt and retried up to 2 times. Each successful batch is upserted to the DB immediately (mid-pipeline persistence). Rate-limit errors halt the loop and write a resume flag file.
Batch size: 5 OpenClaw AI (timeout 1800s) Zod Validation Max 2 retries Rate-limit resume
3
Ingest to Database
Validated AI review results upserted into ReviewedPatch table via Prisma. Unique key: [vendor, issueId]. Updates existing records if already present. Logs written to PipelineRun record for audit trail.
Prisma upsert ReviewedPatch table PipelineRun log
4
Passthrough Safety Net
Any patches skipped by the AI (e.g., timeout, batch failure) are auto-inserted with default values: criticality="Important", decision="Pending". Guarantees zero data loss. Configurable per product via passthrough.enabled.
Zero data loss Default: Important/Pending
5
Auto Archive (Quarterly)
After all 9 products complete review, auto-archive checks if quarterly snapshot should be created. Exports entire ReviewedPatch table → patches.json + metadata.json. Stored at ~/.openclaw/workspace/skills/patch-review/quarterly-archive/{quarter}/. Triggered by POST /api/archive/quarterly/auto-check after each product completes.
Quarterly trigger JSON snapshot Audit trail
RAG Exclusion Strategies (Prompt Injection Prevention)
Prompt Injection Strategy
Products: Red Hat, Oracle, Ubuntu
Samples 3 lines from similar historical patches using query_rag.py and injects them into prompt as "examples to ignore"
File Hiding Strategy
Products: Windows, Ceph, MariaDB, SQL Server, PostgreSQL
Hides patch JSON file from workspace directory, forces AI to use only inline [BATCH DATA]
No Strategy
Products: VMware vSphere
Relies on clear prompt boundaries and generic instruction set without RAG exclusion

Zod Output Schema — ReviewItemSchema
Every AI-returned object must satisfy this schema
IssueID string — patch identifier
Component string — affected software
Version string — version string
Criticality Critical / Important / Moderate / Low
Description string — English summary
KoreanDescription string — Korean summary
Decision optional — Done / Exclude / Pending
Reason optional — justification string
Validation mode: exact — array length must equal batch size;
nonEmpty — used for version-grouped products (e.g. Windows).

Completed batches are persisted to aiReportFile (JSON) and to SQLite after each batch. On resume, already-reviewed issue IDs are loaded into a Set<string> and skipped, so only the remaining patches are re-submitted to the AI.
Debug Artifacts Written Per Run
debug_prompt_{n}.txt
Full prompt text sent to OpenClaw for each batch (including RAG exclusion rules). Written to skillDir/ before each AI call.
aiReportFile (JSON)
Cumulative list of all reviewed patches written after each successful batch. Serves as checkpoint for rate-limit resume.
.rate_limit_flag
Created when a rate-limit error is detected. Checked by the retry endpoint to enable resume mode. Deleted on successful pipeline completion.
