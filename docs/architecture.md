# Patch Review Dashboard V2 Architecture

This document describes the foundational architecture of the Patch Review Dashboard V2, detailing the components, data stores, and backend pipelines. **All information here reflects the factual, operational infrastructure currently running on the server without mock assumptions.**

---

## 1. High-Level Architecture Overview

The system is separated into three primary tiers:

1. **Frontend Presentation Layer** (Next.js App Router)
2. **Backend API & Orchestration Layer** (Next.js APIs + Local Shell execution)
3. **Data Collection & AI Processing Pipeline** (Node.js, Bash, Python, and OpenClaw)

This decoupled approach ensures that heavy data parsing, RAG filtering, and AI evaluations do not block the Web UI, which simply triggers and observes the progress of the underlying OS pipelines using Server-Sent Events (SSE).

---

## 2. Component Details

### 2.1 Web Dashboard (Next.js)
- **Role:** The primary user interface.
- **Functions:** 
  - Displays product matrices (Linux variants, MariaDB, Ceph).
  - Triggers pipeline jobs via `/api/pipeline/execute`.
  - Polls or streams `/api/pipeline/status` to visualize background tasks.
  - Exposes endpoints to finalize AI inputs into SQLite `ReviewedPatch` records.

### 2.2 Relational Database (Prisma + SQLite)
The system uses SQLite, managed via Prisma ORM (`prisma/patch-review.db`), operating with 5 primary entities:

- `PipelineRun`: Tracks metadata regarding current execution runs.
- `RawPatch`: Caches raw JSON responses fetched directly from vendor endpoints.
- `PreprocessedPatch`: Normalized datasets generated by `patch_preprocessing.py`, ready for AI consumption.
- `ReviewedPatch`: The finalized, AI-reviewed or human-approved security recommendations.
- `UserFeedback`: Historical exclusion feedback from security operators used to adjust the RAG database dynamically.

### 2.3 Execution Pipeline (`patch-review` Skill)
The backend delegates all heavy lifting to the `~/.openclaw/workspace/skills/patch-review` directory.
When `/api/pipeline/execute` is called:
1. A Node child process starts running within the `os/linux-v2` environment.
2. It executes vendor-specific scripts (e.g., `rhsa_collector.js`, `oracle_collector.sh`).
3. It consolidates outputs using `patch_preprocessing.py`.
4. It calls `query_rag.py` to retrieve `UserFeedback` exclusion contexts based on similarity searches to prevent repetitive patch approvals for known exceptions.
5. It invokes `openclaw agent:main` to run LLM Impact Analysis based on `SKILL.md` instructions.
6. The exact output is rigorously validated using Zod schemas (`ReviewSchema`). If invalid, a self-healing ping-pong mechanism loops the AI up to 2 retry attempts.

---

## 3. Automation and Triggers

- **Dashboard UI:** Manual trigger execution.
- **REST APIs:** The `trigger.sh` script invokes `http://localhost:3000/api/pipeline/execute`.
- **CRON Schedules:** The server operates `run_collectors_cron.sh` autonomously on the third Sunday of March, June, September, and December (`0 6 15-21 3,6,9,12 * test $(date +\%w) -eq 0`) to maintain a zero-touch operational lifecycle.

---

## 4. File Output Generation
Rather than tightly coupling AI to the database, the AI agent's sole output target is exactly one file: `patch_review_ai_report.json`. This flat JSON array structure ensures atomic transactions; only when the LLM outputs a perfect JSON array does the dashboard commit the records into the database via the `/api/pipeline/finalize` process.
