📅 Data Collection — Cron Schedule & Flow

🕐 Quarterly Collection Timeline

Runs on the 3rd Sunday of March, June, September, December each year. Cron condition: test $(date +%w) -eq 0 ensures Sunday execution within the 15–21 day window.
02:00
SQL Server Collection
Script: run_sqlserver_collector.sh — Collects from Microsoft security update catalog. Windows/SQLServer use aiBatchSize=15 for cross-version comparison.
03:00
Windows Server Collection
Script: run_windows_collector.sh — Windows Server patches. Preprocessing uses grouping removal + aiBatchSize approach (교훈35).
04:00
VMware vSphere Collection
Script: run_vsphere_collector.sh — VMware security advisories (VMSA). vSphere uses individual create instead of createMany for DB ingest.
05:00
PostgreSQL Collection
Script: run_pgsql_collector.sh — PostgreSQL security advisories. File prefix: PGSL- (raw data file prefix convention).
06:00
Linux (RHEL / Oracle Linux / Ubuntu)
Script: run_collectors_cron.sh — All three Linux distributions collected in parallel. 180-day collection window. Review window: 90–180 days (kernel: 0–180 days).
08:00
MariaDB Collection
Script: run_collectors_cron.sh — MariaDB security releases. Uses file-hiding + prompt-injection (both) RAG strategy.
10:00
Ceph Collection
Script: run_collectors_cron.sh — Ceph storage security advisories. csvBOM=true for Korean Excel compatibility.
00:00 (Jan/Apr/Jul/Oct 1st)
Auto Quarterly Archive
POST /api/archive/quarterly/cron — Snapshots all finalized ReviewedPatch records into quarterly archive files.

📋 Cron Expression Details

# Standard quarterly cron pattern (runs 02:00–10:00 on 3rd Sunday of Mar/Jun/Sep/Dec) 0 2 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_sqlserver_collector.sh 0 3 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_windows_collector.sh 0 4 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_vsphere_collector.sh 0 5 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_pgsql_collector.sh 0 6 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_collectors_cron.sh # Linux 0 8 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_collectors_cron.sh # MariaDB 0 10 15-21 3,6,9,12 * test $(date +%w) -eq 0 && /home/citec/run_collectors_cron.sh # Ceph # Quarterly archive (1st of Jan/Apr/Jul/Oct at midnight) 0 0 1 1,4,7,10 * curl -X POST http://localhost:3001/api/archive/quarterly/cron # All collector logs: # /home/citec/patch_collector_cron.log
The test $(date +%w) -eq 0 trick: cron doesn't support "nth weekday of month" natively. This runs the cron daily on days 15–21 but only executes if it's Sunday (day 0).

Stage 1: Preprocessing (Python)

Skipped entirely if isAiOnly=true. If preprocessing script is missing, pipeline fails IMMEDIATELY (no fallback).
# Standard Linux preprocessor python3 patch_preprocessing.py --vendor redhat --days 180 python3 patch_preprocessing.py --vendor oracle --days 180 python3 patch_preprocessing.py --vendor ubuntu --days 180 # Windows / SQL Server preprocessor (with date window) python3 windows_preprocessing.py --days 180 --days_end 90 # Output: patches_for_llm_review_{vendor}.json
Log Tag Pattern (triggers dashboard count update):
[REDHAT-PREPROCESS_DONE] count=42 [ORACLE-PREPROCESS_DONE] count=17 [UBUNTU-PREPROCESS_DONE] count=23
Normalization Steps:
  • ▸ Parse raw JSON from collector output
  • ▸ Apply date window filtering (180 days collection, 90–180 review)
  • ▸ Kernel exception: kernel packages reviewed for full 0–180 day range
  • ▸ Normalize fields to LLM-friendly format
  • ▸ Store to PreprocessedPatch table via Prisma upsert
  • ▸ Emit [VENDOR-PREPROCESS_DONE] count= log tag

Stage 2: RAG Exclusion Setup (2 Strategies)

Strategy A: prompt-injection
Applied to: RHEL, Oracle, Ubuntu, vSphere (and all others as of 2026-03-19)
  • ▸ Runs os/linux/query_rag.py to query ChromaDB
  • ▸ Source: user_exclusion_feedback.json
  • ▸ Injects CRITICAL INSTRUCTION block into AI prompt
  • ▸ cwd always fixed to os/linux/ (shared RAG directory)
Strategy B: file-hiding
Applied to: Windows, Ceph, MariaDB, SQL Server, PostgreSQL
  • ▸ Before AI: rename normalized/ → normalized_hidden/
  • ▸ Before AI: rename patches_for_llm_review_*.json → *.hidden
  • ▸ After AI: restore original names (even on exception — finally block)
  • ▸ Purpose: prevent OpenClaw from reading previous review files (bias prevention)
Combined strategy type: 'both' applies file-hiding first, then prompt-injection. Products: Windows, Ceph, MariaDB, SQL Server, PostgreSQL all use type: 'both'.
// query_rag.py invocation (cwd always os/linux/) python3 os/linux/query_rag.py \ --query "security patch exclusion reasons" \ --n_results 10 // Result injected as CRITICAL INSTRUCTION block into prompt

Stage 3: AI Review Loop (runAiReviewLoop)

prunePatch (all products)
  • ▸ URL → [URL] replacement
  • ▸ Text truncated to 5000 chars
prunePatchLinux (Linux only)
  • ▸ All above, plus:
  • ▸ Array fields (packages, cves, etc.) limited to max 15 items
gateway closed (1000): Does NOT immediately reject — waits for full response. Only enters retry if the extracted JSON is invalid. (교훈16)
Rate limit (429): Creates /tmp/.rate_limit_{productId} file → next run enters resume mode (skips preprocessing, skips completed batches)

Stage 4: DB Ingest (ingestToDb)

// If not resume/aiOnly: DELETE existing records first await prisma.reviewedPatch.deleteMany({ where: { productId } }); // Filter out: decision='Exclude' OR criticality in [moderate, medium, low] // These are NOT inserted to ReviewedPatch const toInsert = results.filter(r => r.decision !== 'Exclude' && !['moderate', 'medium', 'low'].includes(r.criticality) ); // Insert (createMany for most products) await prisma.reviewedPatch.createMany({ data: toInsert }); // vSphere exception: uses individual create() calls for (const item of toInsert) { await prisma.reviewedPatch.create({ data: item }); }
ReviewedPatch.issueId is @unique — this enforces deduplication at the DB level. If the same issueId appears twice, the second upsert will fail gracefully.

Stage 5: Passthrough Safety Net (runPassthrough)

// Find patches in PreprocessedPatch NOT yet in ReviewedPatch const missing = await prisma.preprocessedPatch.findMany({ where: { productId, issueId: { notIn: reviewedIds } } }); // Insert with default values (safety net) await prisma.reviewedPatch.createMany({ data: missing.map(p => ({ ...p, criticality: 'Important', decision: 'Pending' })) });

🔒 withOpenClawLock — Concurrency Control

// Directory mutex — prevents concurrent AI calls // Lock file: /tmp/openclaw_execution.lock async function withOpenClawLock<T>(fn: () => Promise<T>): Promise<T> { const lock = await acquireLock('/tmp/openclaw_execution.lock'); try { return await fn(); } finally { releaseLock(lock); } }
🤖 AI Review System — Deep Dive

⚡ OpenClaw AI Agent

# Full invocation signature openclaw agent:main \ --json \ --timeout 1800 \ --session-id {unique-per-batch} \ -m {prompt-string}
  • ▸ --json: Forces JSON-only output mode
  • ▸ --timeout 1800: 30-minute timeout per batch
  • ▸ --session-id {unique}: Unique per batch to prevent cross-batch contamination
  • ▸ -m {prompt}: Full prompt string including RAG context
  • ▸ Spawned as child process by queue.ts worker
  • ▸ Protected by withOpenClawLock (directory mutex)

🔄 Error Handling & Retry Logic

Gateway Closed (code 1000)
  • ▸ Does NOT immediately reject (교훈16)
  • ▸ Waits for full AI response to complete
  • ▸ Only enters retry if extracted JSON is invalid
  • ▸ Valid JSON → accepted even if gateway closes
Rate Limit (HTTP 429)
  • ▸ Creates /tmp/.rate_limit_{productId}
  • ▸ Next pipeline run detects rate limit file
  • ▸ Enters resume mode automatically
  • ▸ Skips preprocessing + completed batches
Zod Validation Failure
  • ▸ Up to 2 retries per batch
  • ▸ Zod error message injected into retry prompt
  • ▸ AI asked to correct its output format
  • ▸ After 2 failures → passthrough safety net
Crash Recovery
  • ▸ Intermediate DB save after each batch
  • ▸ Resume mode skips completed batches
  • ▸ Progress preserved across crashes

🙈 File-Hiding Strategy — Bias Prevention

// Before AI review (hide previous review files) fs.renameSync('normalized/', 'normalized_hidden/'); fs.renameSync( `patches_for_llm_review_${vendor}.json`, `patches_for_llm_review_${vendor}.json.hidden` ); try { // Run AI review await runAiReviewLoop(...); } finally { // ALWAYS restore — even if AI throws exception fs.renameSync('normalized_hidden/', 'normalized/'); fs.renameSync( `patches_for_llm_review_${vendor}.json.hidden`, `patches_for_llm_review_${vendor}.json` ); }
Purpose: OpenClaw agent has file-reading tools. Without hiding, it could read the previous quarter's review files and be biased toward the same decisions. File-hiding forces fresh evaluation.
👤 Admin Review & Feedback Loop

📋 Admin Review Workflow — Step by Step

1
Pipeline Runs
Admin triggers pipeline from dashboard. BullMQ processes job. Real-time logs via SSE. Patches appear in ReviewedPatch table with AI-assigned criticality and decision.
2
Review Patches on Dashboard
Admin views product detail page. Table with filters (by criticality, decision, component). Each patch shows: IssueID, Component, Version, Criticality, AI Decision, Korean Description, Reason.
3
Provide Exclusion Feedback
Admin marks patches for exclusion patterns. Feedback text explains why similar patches should be excluded in future runs. Stored in UserFeedback table AND user_exclusion_feedback.json.
4
Feedback → ChromaDB
user_exclusion_feedback.json is embedded into ChromaDB vector store. Future RAG queries retrieve semantically similar past exclusions. Circular feedback loop improves over time.
5
Finalize — Generate CSV
Admin clicks Finalize. POST /api/pipeline/{product}/finalize generates final_approved_patches_{product}.csv. Non-Linux products use UTF-8 BOM (csvBOM=true) for Excel Korean text compatibility.

📊 Prisma Schema — 5 Models

Model: RawPatch — Cached raw JSON from collectors▶
model RawPatch { id Int @id @default(autoincrement()) productId String issueId String rawJson String // full raw JSON from collector createdAt DateTime @default(now()) updatedAt DateTime @updatedAt @@unique([productId, issueId]) }
Model: PreprocessedPatch — Python-normalized patches▶
model PreprocessedPatch { id Int @id @default(autoincrement()) productId String issueId String component String version String vendor String date String description String osVersion String? // passthrough field rawData String // normalized JSON for LLM createdAt DateTime @default(now()) @@unique([productId, issueId]) }
Model: ReviewedPatch — AI+Admin final approved patches▶
model ReviewedPatch { id Int @id @default(autoincrement()) productId String issueId String @unique // deduplication key component String version String vendor String date String criticality String // Important|Critical|Moderate|Low description String koreanDescription String decision String // Include|Exclude|Pending reason String osVersion String? finalCsvFile String? // path to exported CSV createdAt DateTime @default(now()) updatedAt DateTime @updatedAt }
issueId is @unique — ensures no duplicate patch IDs. Filters at ingest: Exclude + moderate/medium/low criticality patches are NOT stored here.
Model: UserFeedback — Admin exclusion feedback (RAG source)▶
model UserFeedback { id Int @id @default(autoincrement()) productId String issueId String feedbackType String // 'exclude' | 'include' reason String // admin's explanation — used for RAG createdAt DateTime @default(now()) } // Also written to: user_exclusion_feedback.json (for ChromaDB)
Model: PipelineRun — Pipeline execution metadata/audit▶
model PipelineRun { id Int @id @default(autoincrement()) productId String jobId String status String // running|completed|failed isRetry Boolean @default(false) isAiOnly Boolean @default(false) startedAt DateTime @default(now()) completedAt DateTime? errorMsg String? }

⚡ SQLite WAL Mode

// Prisma singleton (prevents HMR duplicates) const globalWithPrisma = global as typeof global & { prisma: PrismaClient | undefined }; export const prisma = globalWithPrisma.prisma ?? new PrismaClient(); if (process.env.NODE_ENV !== 'production') globalWithPrisma.prisma = prisma; // WAL mode enables dashboard reads during pipeline writes // PRAGMA journal_mode=WAL;
WAL (Write-Ahead Log) mode allows concurrent reads during pipeline execution. Without WAL, SQLite would lock the database during AI review writes, blocking the dashboard.
