# 요구사항 정의서 — patch-review-dashboard-v2

> 이 문서만으로 patch-review-dashboard-v2와 동등한 시스템을 처음부터 구축할 수 있도록 작성되었다.

---

## 1. 프로젝트 개요

### 1.1 배경 및 목적

기업 IT 인프라에서 운영 중인 9개 핵심 제품(OS, DB, Storage, Virtualization)에 대해 분기마다 보안 패치를 체계적으로 수집·평가·승인하는 내부 관리 시스템이 필요하다. 기존에는 담당자가 각 제품의 보안 공지 사이트를 수동으로 확인했으나 다음 문제가 있었다:

- 9개 제품 × 수십 건의 분기 패치를 수동 검토하면 수 일이 소요됨
- 검토 기준이 담당자마다 달라 일관성 부족
- 검토 이력이 관리되지 않아 과거 결정을 재사용할 수 없음
- 최종 승인 결과를 CSV로 만드는 작업도 수동으로 진행

이를 해결하기 위해 **AI 기반 자동 패치 리뷰 대시보드**를 구축한다.

### 1.2 시스템 한 줄 요약

> 분기별 보안 패치를 수집 → Python 전처리 → OpenClaw AI 자동 평가 → 관리자 최종 승인 → CSV 출력하는 웹 대시보드

## 3. 기능 요구사항

### 3.1 데이터 수집 (FR-01)

- **FR-01-1**: 각 제품의 보안 공지 API/피드를 크론으로 분기별 자동 수집해야 한다.
- **FR-01-2**: 수집된 원시 데이터는 JSON 파일로 저장된다. 파일명은 `{PREFIX}-{ID}.json` 형식이다.
- **FR-01-3**: 수집 일정: 분기의 세 번째 일요일 06:00 (크론 스케줄러).
- **FR-01-4**: 수집 범위: 최근 180일치 어드바이저리.
- **FR-01-5**: 데이터 파일은 각 제품 스킬 디렉토리의 `{product}_data/` 하위에 저장된다.

### 3.2 파이프라인 실행 (FR-02)

- **FR-02-1**: 관리자가 웹 UI에서 제품별로 파이프라인을 실행할 수 있어야 한다.
- **FR-02-2**: 파이프라인 실행 시 실시간 로그가 브라우저에 스트리밍되어야 한다(SSE).
- **FR-02-3**: 동시에 두 개 이상의 파이프라인이 실행되면 큐에서 순차 처리한다.
- **FR-02-4**: 파이프라인은 전처리 단계를 건너뛰고 AI 리뷰만 재실행하는 옵션(isAiOnly)을 지원해야 한다.
- **FR-02-5**: API Rate Limit 발생 시 중단된 지점부터 재개(Resume Mode)할 수 있어야 한다.

### 3.3 전처리 (FR-03)

- **FR-03-1**: 각 제품의 Python 전처리 스크립트가 원시 JSON을 정규화된 구조로 변환해야 한다.
- **FR-03-2**: 전처리 결과는 `patches_for_llm_review_{vendor}.json` 파일로 저장된다.
- **FR-03-3**: Windows/SQL Server는 날짜 윈도우(`--days 180 --days_end 90`)를 사용해 최근 90일을 제외한 패치만 포함한다.
- **FR-03-4**: Linux 제품은 비커널 컴포넌트의 경우 발행일 기준 90~180일 범위만 검토한다. 커널(`kernel`, `kernel-uek`, `linux-image-*`)은 0~180일 전체 포함.
- **FR-03-5**: 전처리 완료 시 `[{LOGTAG}-PREPROCESS_DONE] count=N` 형식 로그를 출력해야 한다.

### 3.4 AI 리뷰 (FR-04)

- **FR-04-1**: OpenClaw AI CLI(`openclaw agent:main`)로 패치를 자동 평가해야 한다.
- **FR-04-2**: 패치는 배치 단위(기본 5개, 제품별 설정 가능)로 순차 처리된다.
- **FR-04-3**: AI는 각 패치에 대해 IssueID, Component, Version, Vendor, Date, Criticality, Description(영문), KoreanDescription(한글)을 반환해야 한다.
- **FR-04-4**: Criticality가 moderate/medium/low이거나 Decision이 Exclude이면 DB에 저장하지 않는다.
- **FR-04-5**: AI가 평가하지 않고 누락한 패치는 패스스루(Passthrough) 안전망으로 `criticality=Important, decision=Pending`으로 자동 삽입된다.
- **FR-04-6**: Zod 스키마 검증 실패 시 최대 2회 재시도한다.

### 3.5 RAG 기반 제외 규칙 (FR-05)

- **FR-05-1**: 관리자가 특정 패치를 수동으로 Exclude 처리할 때 이유를 함께 등록할 수 있어야 한다.
- **FR-05-2**: 등록된 배제 이유는 ChromaDB에 저장되고, 이후 AI 리뷰 시 유사 패치를 자동으로 Exclude 처리하도록 프롬프트에 주입된다(prompt-injection 방식).
- **FR-05-3**: OpenClaw 파일 도구가 이전 리뷰 파일에 접근해 편향을 만드는 것을 방지하기 위해, 비Linux 5개 제품은 AI 호출 전 파일을 임시로 숨긴다(file-hiding 방식).

### 3.6 관리자 최종 승인 (FR-06)

- **FR-06-1**: 관리자가 AI 리뷰 결과를 검토하고 최종 승인(Finalize)할 수 있어야 한다.
- **FR-06-2**: Finalize 시 승인된 패치 목록을 CSV 파일로 저장한다.
- **FR-06-3**: Windows Server, MariaDB, SQL Server, PostgreSQL, VMware vSphere의 CSV는 UTF-8 BOM을 포함해야 한다(Excel 한글 깨짐 방지).
- **FR-06-4**: Finalize 완료 후 모든 제품이 완료 상태이면 분기 아카이브가 자동 생성된다.

### 3.7 데이터 조회 및 내보내기 (FR-07)

- **FR-07-1**: 카테고리별, 제품별 패치 현황(수집/전처리/리뷰/승인 단계별 카운트)을 대시보드 메인에 표시한다.
- **FR-07-2**: 관리자는 제품별 또는 카테고리 전체 패치를 CSV로 다운로드할 수 있다.
- **FR-07-3**: 각 파이프라인 단계(Raw/Preprocessed/Reviewed)의 JSON 데이터를 모달로 조회할 수 있다.

### 3.8 분기 아카이브 (FR-08)

- **FR-08-1**: 분기 완료 후 전체 ReviewedPatch를 스냅샷으로 저장한다.
- **FR-08-2**: 아카이브는 `quarterly-archive/Q{N}-{YEAR}/metadata.json`과 `patches.json`으로 구성된다.
- **FR-08-3**: 과거 분기 아카이브를 웹 UI에서 조회할 수 있어야 한다.

### 3.9 국제화 (FR-09)

- **FR-09-1**: UI는 한국어(기본)와 영어를 지원해야 한다.
- **FR-09-2**: 언어 전환은 쿠키(`NEXT_LOCALE`) 기반으로 동작하며 페이지 새로고침 없이 전환된다.

## 4. 비기능 요구사항

### 4.1 성능 (NFR-01)

- 파이프라인 실행 중에도 대시보드 조회(읽기)가 가능해야 한다. → SQLite WAL 모드로 해결.
- AI 리뷰 루프는 중단 시 재개 가능해야 하며, 배치 완료마다 중간 저장이 이루어져야 한다.
- 단일 AI 호출 타임아웃: 1800초.

### 4.2 안정성 (NFR-02)

- 여러 제품 파이프라인이 동시에 실행될 경우 AI 호출이 직렬화되어야 한다(글로벌 뮤텍스).
- 파이프라인 실패 시 BullMQ 상태를 Reset 기능으로 복구할 수 있어야 한다.
- file-hiding 방식에서 예외 발생 시에도 `finally` 블록에서 파일을 반드시 복원해야 한다.

### 4.3 확장성 (NFR-03)

- 신규 제품 추가 시 `products-registry.ts`에 `ProductConfig` 1개 추가 + API route 2개(run/finalize) 생성만으로 완료되어야 한다.
- Python 전처리 스크립트는 각 제품의 스킬 디렉토리에 독립적으로 위치한다.

### 4.4 운영 (NFR-04)

- 서버 재부팅 시 자동으로 서비스가 재시작되어야 한다(PM2 + systemd).
- 빌드 실패 시 현재 서비스가 중단되지 않아야 한다.
- 배포 포트: 3001.

### 4.5 보안 (NFR-05)

- 관리자 전용 내부 네트워크 서비스이므로 별도 인증은 구현하지 않는다.
- Python venv는 Next.js 프로젝트 디렉토리 내에 존재해서는 안 된다(Turbopack symlink 빌드 오류 방지).

| US-01 | 담당자 | 대시보드 홈에서 각 제품의 파이프라인 진행 상태를 한눈에 보고 싶다 | 카테고리별 제품 카드에 수집/전처리/리뷰/승인 카운트가 표시됨 |

| US-04 | 담당자 | 특정 패치를 다음 분기에도 자동으로 제외되도록 등록하고 싶다 | 피드백 등록 후 이후 파이프라인에서 해당 패치 Exclude 처리 |
| US-05 | 담당자 | 승인된 패치를 CSV로 다운로드하고 싶다 | Export 버튼 클릭 시 카테고리/제품별 CSV 다운로드 |
| US-06 | 담당자 | 과거 분기의 리뷰 결과를 조회하고 싶다 | 아카이브 페이지에서 분기별 패치 목록 조회 가능 |

| AI CLI | OpenClaw CLI (`openclaw agent:main`)가 서버에 설치되어야 함 |
