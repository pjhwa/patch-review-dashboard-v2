### 1.2 핵심 설계 결정

| 결정 | 이유 |
|------|------|
| BullMQ + Redis 큐 | 파이프라인이 오래 걸리므로 HTTP 요청 밖에서 비동기 실행 필요 |
| SQLite + WAL 모드 | 단일 서버 운영에 적합, WAL로 읽기-쓰기 동시 접근 허용 |
| Next.js App Router | SSR + Server Components로 초기 로딩 최적화 |
| SSE 실시간 스트리밍 | WebSocket 없이 단방향 실시간 로그 전달 |
| 중앙 제품 레지스트리 | `ProductConfig` 1개 추가만으로 신규 제품 온보딩 |
| 글로벌 뮤텍스(mkdirSync) | OS 레벨 원자적 연산으로 AI 동시 호출 방지 |

## 2. 중앙 제품 레지스트리 설계

### 2.1 ProductConfig 인터페이스

모든 제품의 파이프라인 동작은 `src/lib/products-registry.ts`의 `ProductConfig` 인터페이스로 표현된다.

```typescript
interface ProductConfig {
    // 기본 식별 정보
    id: string;              // 라우팅/DB 조회용 식별자
    name: string;            // UI 표시명
    vendorString: string;    // DB vendor 필드값 (preprocessedPatchMapper와 반드시 일치)
    category: 'os' | 'storage' | 'database' | 'virtualization';
    active: boolean;

    // 파일시스템 경로
    skillDirRelative: string;    // ~/.openclaw/.../patch-review/ 기준 상대경로
    dataSubDir: string;          // skillDir 아래 원시 수집 데이터 디렉토리
    rawDataFilePrefix: string[]; // 수집 파일 식별 prefix

    // 전처리
    preprocessingScript: string; // Python 스크립트 파일명
    preprocessingArgs: string[]; // CLI 인수

    // 파일 경로 (모두 skillDir 기준)
    patchesForReviewFile: string; // AI 리뷰용 JSON
    aiReportFile: string;         // AI 리뷰 결과 JSON
    finalCsvFile: string;         // 최종 승인 CSV

    // BullMQ
    jobName: string;              // 큐에서 분기 식별자
    rateLimitFlag: string;        // Rate Limit 시 생성되는 플래그 파일
    logTag: string;               // 로그 접두어 (예: 'REDHAT')

    // AI 프롬프트 설정
    aiEntityName: string;
    aiVendorFieldValue: string;
    aiComponentDefault: string;
    aiVersionGrouped: boolean;    // 버전 그룹 방식 여부 (SQL Server)
    aiBatchValidation: 'exact' | 'nonEmpty';
    aiBatchSize?: number;         // 제품별 배치 크기 (기본: 5)

    // RAG 제외 전략
    ragExclusion?: {
        type: 'file-hiding' | 'prompt-injection' | 'both';
        normalizedDirName?: string;  // file-hiding 대상 디렉토리
        queryScript?: string;        // RAG 조회 스크립트
        queryTextSampleSize?: number;
    };

    // 패스스루 안전망
    passthrough: {
        enabled: boolean;
        fallbackCriticality: string;
        fallbackDecision: string;
    };

    // 함수형 설정
    collectedFileFilter: (filename: string) => boolean;
    preprocessedPatchMapper: (raw: any) => object;  // raw JSON → DB 스키마
    csvBOM: boolean;  // Excel 한글 깨짐 방지용 BOM
    buildPrompt: (skillDir: string, batchSize: number, prunedBatch: any[]) => string;
}
```

### 2.2 제품별 RAG 전략

| 제품 | RAG 전략 | 이유 |
|------|---------|------|
| redhat, oracle, ubuntu, vsphere | `prompt-injection` | normalized/ 디렉토리 없음 |
| windows, ceph, mariadb, sqlserver, pgsql | `both` | file-hiding + prompt-injection 병행 |

- **file-hiding**: AI 호출 전 `normalized/` 디렉토리와 `patches_for_llm_review_*.json`을 `*_hidden`으로 임시 변경 → AI 완료 후 복원
- **prompt-injection**: `query_rag.py`로 ChromaDB에서 과거 배제 이유 조회 → AI 프롬프트에 `CRITICAL INSTRUCTION` 블록으로 주입
- `query_rag.py`는 항상 `os/linux/` 공유 디렉토리를 cwd로 실행 (모든 제품 공통)

## 3. 파이프라인 5단계 흐름

### 3.1 단계별 상세

```
[UI] POST /api/pipeline/{product}/run
    ↓ pipelineQueue.add(jobName, { isRetry, isAiOnly })
    ↓ 응답: { jobId }
    ↓ 클라이언트: GET /api/pipeline/stream?jobId=...

[BullMQ Worker] runProductPipeline(job, productCfg)

Stage 1: 전처리 (isAiOnly=false일 때만)
  - python3 {preprocessingScript} {preprocessingArgs}
  - cwd: skillDir
  - 출력: patches_for_llm_review_{vendor}.json
  - 로그: [LOGTAG-PREPROCESS_DONE] count=N
  - 실패 시 파이프라인 즉시 종료

Stage 2: RAG 제외 규칙 로드 (prompt-injection 또는 both)
  - python3 query_rag.py "{queryTextContext}"
  - cwd: os/linux/ 공유 디렉토리 (고정)
  - 결과: ragExclusions 문자열 (프롬프트에 삽입)

Stage 3: AI 리뷰 루프 (runAiReviewLoop)
  for i in range(0, len(patches), BATCH_SIZE):
    1. cleanupSessions() — sessions.json 포함 삭제
    2. file-hiding 적용 (해당 제품만)
    3. buildPrompt(skillDir, batchSize, prunedBatch)
    4. openclaw agent:main --json --timeout 1800 --session-id {unique} -m {prompt}
       → withOpenClawLock 내에서 실행 (글로벌 뮤텍스)
    5. extractJsonArray() — 코드펜스 제거 후 JSON 배열 추출
    6. Zod 검증 (ReviewItemSchema)
    7. 검증 실패 시 최대 2회 재시도
    8. 배치 완료 시 aiReportFile에 중간 저장
  → file-hiding 복원 (finally 블록)

Stage 4: DB 반영 (ingestToDb)
  - isAiOnly=false이면: vendor별 기존 레코드 deleteMany 후 재삽입
  - Decision='Exclude' 또는 criticality가 moderate/medium/low → 삽입 안 함
  - vsphere: createMany 대신 개별 create (ID 충돌 방지)
  - 삽입 성공: aiReviewedIds에 추가

Stage 5: 패스스루 안전망 (runPassthrough)
  - passthrough.enabled=true인 제품만 실행
  - PreprocessedPatch에서 AI가 미처리한 항목 찾기
  - fallbackCriticality='Important', fallbackDecision='Pending'으로 upsert
```

### 3.2 Resume Mode (재개 모드)

Rate Limit 발생 시 자동 활성화:

```
1. stderr에 'rate limit' 포함 → rateLimitFlag 파일 생성 (/tmp/.rate_limit_{productId})
2. 다음 파이프라인 실행:
   - isAiOnly=true 또는 isRetry=true && rateLimitFlag 존재 → isResumeMode=true
   - aiReportFile에서 이미 완료된 배치 읽기
   - 완료된 배치는 건너뜀
3. rateLimitFlag 파일을 삭제 후 재개
```

## 4. 데이터베이스 설계

### 4.1 Prisma 스키마 (SQLite)

```prisma
// 수집된 원시 데이터 캐싱 (현재는 파일시스템 활용, 미래 확장용)
model RawPatch {
  id            String   @id @default(uuid())
  vendor        String
  originalId    String
  data          String   // JSON string
  collectedAt   DateTime @default(now())
  pipelineRunId String?
  preprocessed  Boolean  @default(false)
}

// 전처리 완료된 정규화 패치 (파이프라인 단계 추적용)
model PreprocessedPatch {
  id                  String   @id @default(uuid())
  vendor              String
  issueId             String   // @unique 아님 — deleteMany+create 패턴
  osVersion           String?
  component           String
  version             String
  severity            String?
  releaseDate         String?
  description         String?
  collectedAt         DateTime @default(now())
  pipelineRunId       String?
  isReviewed          Boolean  @default(false)
  isAiReviewRequested Boolean  @default(false)
}

// AI + 관리자 최종 승인 패치
model ReviewedPatch {
  id                String   @id @default(uuid())
  vendor            String
  issueId           String   @unique  // ← 유일성 보장, upsert 키
  osVersion         String?
  component         String
  version           String
  criticality       String
  description       String
  koreanDescription String
  decision          String   // Done | Pending | Exclude
  reason            String?
  reviewedAt        DateTime @default(now())
  pipelineRunId     String?
}

// RAG 시스템용 관리자 배제 이력
model UserFeedback {
  id              String   @id @default(uuid())
  issueId         String
  vendor          String
  component       String
  version         String
  userReason      String
  createdAt       DateTime @default(now())
}

// 파이프라인 실행 메타데이터
model PipelineRun {
  id          String    @id @default(uuid())
  status      String    // running, completed, error
  message     String
  startedAt   DateTime  @default(now())
  completedAt DateTime?
  logs        String?
}
```

### 4.2 WAL 모드 활성화


파이프라인 실행 중에도 대시보드 읽기 조회가 가능하도록 WAL 모드를 사용한다.

## 6. 프론트엔드 구조

### 6.1 페이지 라우팅

```
src/app/
  page.tsx                           ← 홈: SSR 통계 + 카테고리 그리드
  layout.tsx                         ← 루트 레이아웃 (Geist 폰트, LanguageToggle)
  category/
    [categoryId]/
      page.tsx                       ← 카테고리 상세 (제품 목록 + ProductGrid)
      [productId]/
        page.tsx                     ← 제품 상세 (서버 컴포넌트 래퍼)
        ClientPage.tsx               ← 클라이언트 컴포넌트 (리뷰 테이블 + Finalize)
        archive/
          page.tsx                   ← 아카이브 목록
          [quarter]/page.tsx         ← 분기별 아카이브 뷰어
```

### 6.2 핵심 컴포넌트

**ProductGrid.tsx** — 파이프라인 실행 및 로그 스트리밍

```
기능:
- 페이지 로드 시 GET /api/pipeline으로 진행 중 잡 자동 감지 → SSE 재연결
- 실행 버튼 클릭 → POST {pipelineRunUrl} → jobId 수신 → SSE 연결
- 로그 버퍼: 최근 30줄만 유지
- 로그 태그 파싱 (generic regex):
  - PREPROCESS_DONE 포함 → 전처리 완료 카운트 갱신
  - /\[\w+-PIPELINE\]|\[\w+-AI\]/ → 단계 진행 메시지
- 확인 다이얼로그: 실행 전 재확인 모달
```

**PremiumCard.tsx** — 단계별 카운터가 있는 제품 카드

```
stages: { collected, preprocessed, reviewed, approved }
isReviewCompleted: true이면 완료 배지 표시
```

**LanguageToggle.tsx** — KO/EN 전환 (NEXT_LOCALE 쿠키)

**StageJSONViewer.tsx** — 단계별 raw JSON 모달 뷰어
