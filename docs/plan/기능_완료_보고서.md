## 3. 아키텍처 핵심 결정 이력

### 3.1 v2 핵심 혁신 — 중앙 제품 레지스트리

v1에서 제품별 파이프라인 로직이 분산되어 있던 것을 v2에서 `ProductConfig` 인터페이스 하나로 통합했다. 이로써:

- 신규 제품 추가 시 변경 파일: `products-registry.ts` + API route 2개 (run/finalize)
- `ProductGrid.tsx`의 로그 태그 파싱: generic regex로 신규 제품 추가 시 코드 수정 불필요
- `export/route.ts`: PRODUCT_REGISTRY 기반 동적 처리

### 3.2 RAG 전략 통합 (2026-03-19)

기존 Linux 전용이었던 prompt-injection RAG를 전 제품으로 확장했다.
- `query_rag.py`의 cwd를 항상 `os/linux/` 공유 디렉토리로 고정
- `isLinux &&` 조건 제거 → `ragExclusion.type === 'prompt-injection' || 'both'` 조건으로 변경

### 3.3 Windows 전처리 리팩터링

기존 버전 그룹 출력(4개 `WINDOWS-GROUP-*`) → 개별 패치 레코드(12개) 출력으로 변경:
- AI가 같은 OS버전 패치를 직접 비교해 최적 1개 선택
- `aiBatchSize: 15` 도입으로 12개 패치가 한 배치에 포함

## 4. 검증된 주요 동작

### 4.1 Rate Limit 처리

```
1. stderr에 'rate limit' 감지
2. /tmp/.rate_limit_{productId} 파일 생성
3. aiReportFile에 완료된 배치까지 저장
4. 파이프라인 종료 (오류로 표시)
5. 재실행: isAiOnly=true → isResumeMode=true
6. 완료된 배치 건너뜀, 이후 배치부터 재개
```

### 4.2 Gateway Closed 처리

```
OpenClaw gateway 연결 실패 → embedded 모드 폴백 → 계속 실행
queue.ts: stderr에 'gateway closed' 검사 제거 (즉시 reject 금지)
프로세스 종료 코드(exit code 0)만으로 성공/실패 판단
```

### 4.3 패스스루 안전망

```
AI 리뷰 완료 후:
1. DB의 PreprocessedPatch WHERE vendor = {vendor} 조회
2. AI가 리뷰한 issueId Set과 비교
3. 누락된 항목 → criticality='Important', decision='Pending'으로 upsert
4. 로그: [PASSTHROUGH] Ingested N passthrough patches.
```

## 7. 교훈 — 이 시스템을 재구축할 때 반드시 알아야 할 것들

### 교훈 T1: ProductConfig vendorString과 preprocessedPatchMapper 일치

`vendorString` (DB vendor 필드)과 `preprocessedPatchMapper`의 `vendor:` 값이 다르면 passthrough, ingest, RAG 조회가 모두 오작동한다. 신규 제품 추가 시 반드시 두 값이 동일한지 교차 검증하라.

### 교훈 T2: cleanupSessions()에서 sessions.json 필수 삭제

`sessions.json`을 삭제하지 않으면 이전 파이프라인 실행의 오염된 세션 컨텍스트로 AI가 응답한다. `.lock`/`.jsonl` 파일만 삭제하면 부족하다.

### 교훈 T3: gateway closed 즉시 reject 금지

`stderr`에 "gateway closed"가 나타나도 즉시 Promise를 reject하지 마라. OpenClaw는 자동으로 embedded 모드로 폴백해 계속 실행된다. 프로세스 exit code만으로 성공/실패를 판단해야 한다.

### 교훈 T4: SKILL.md는 100줄 이상 필수

SKILL.md가 100줄 미만이면 OpenClaw embedded 모드에서 "Offset 100 is beyond end of file" 오류로 배치가 영구 실패한다. 또한 `## 4.` 섹션이 반드시 있어야 한다 (프롬프트가 "section 4" 참조).

### 교훈 T5: file-hiding은 finally 블록에서 반드시 복원

예외 발생 시에도 숨긴 파일이 복원되지 않으면 다음 파이프라인 실행에서 AI가 입력 데이터를 읽지 못한다. try-finally 패턴으로 복원을 보장해야 한다.

### 교훈 T6: SQL Server/Windows CVE faq 필드 제거

Microsoft 계열 데이터의 CVE 객체에 있는 `faq` 필드는 2000자 이상의 반복 텍스트다. 이를 프롬프트에 포함하면 컨텍스트 한도를 초과해 gateway closed가 발생한다.

### 교훈 T7: query_rag.py cwd는 항상 os/linux/ 고정

모든 제품의 RAG prompt-injection에서 `query_rag.py`는 `os/linux/` 공유 디렉토리를 cwd로 실행해야 한다. `user_exclusion_feedback.json`과 `chroma_db/`가 이 위치에 있기 때문이다.

### 교훈 T8: Python venv는 patch-review/ 내부 금지

Turbopack이 venv의 절대경로 symlink를 만나면 빌드 패닉이 발생한다. Python 가상환경은 `~/.openclaw/workspace/` 외부에 두거나 `.gitignore`에 `**/venv` 추가 후 삭제한다.

### 교훈 T9: preprocessedPatchMapper 필드명은 실제 JSON과 반드시 일치

`p.id` vs `p.patch_id`, `p.date` vs `p.issued_date` 등 한 곳이라도 다르면 `ingestToDb`에서 `issueId: undefined`로 DB 삽입이 조용히 실패한다. 증상은 "AI 리뷰 N개인데 전처리 카운트 0".

### 교훈 T10: Windows aiBatchSize는 전체 패치 수보다 크게

Windows 패치가 12개라면 `aiBatchSize: 15`로 설정해야 한 배치에 전체가 들어가 같은 OS버전 패치를 AI가 비교할 수 있다. BATCH_SIZE=5이면 12개가 3 배치로 분리되어 버전 간 비교가 불가능하다.

### 교훈 T11: GitHub에 심볼릭 링크로 patch-review/ 연결 금지

`ln -sf ~/.openclaw/... patch-review`로 심볼릭 링크를 만들면 git이 파일 내용 대신 링크 경로 문자열만 추적한다. pre-push hook + rsync 패턴을 사용하라.

### 교훈 T12: gitignore에 **/archive 설정 시 API 경로 예외 추가 필수

`.gitignore`의 `**/archive`가 `src/app/api/archive/`도 차단한다. `!src/app/api/archive` 예외를 반드시 추가하라.

### 교훈 T13: 분기 아카이브 자동 생성 — client-side fire-and-forget

finalize 라우트 7개를 모두 수정하는 대신, `ClientPage.tsx`의 finalize 성공 직후 `POST /api/archive/quarterly/auto-check`를 fire-and-forget으로 호출한다. 변경 범위가 1개 파일로 최소화된다.

### 교훈 T14: BullMQ Worker는 Next.js 앱과 동일 프로세스에서 실행

`queue.ts`의 Worker는 별도 프로세스가 아닌 Next.js 앱 서버에서 시작된다. `startWorker()` 호출을 앱 초기화 시점에 보장해야 하며, `workerStarted` 플래그로 중복 실행을 방지한다.

### 교훈 T15: patch-review/scripts/queue.ts도 Next.js 빌드에 포함됨

`src/lib/queue.ts`가 아닌 `patch-review/scripts/queue.ts`도 TypeScript 빌드 대상이다. 두 파일 모두 타입 에러 없이 유지해야 한다.
