### 2026-03-06 Issue Breakdown
- **Issue**: The 'Retry AI Review' (manual-review or isAiOnly) pipeline hung at 0% progress indefinitely on the dashboard.
- **Root Cause**: The BullMQ worker (queue.ts) was running `openclaw ask` which does not emit the necessary stdout logging patterns (e.g. `[LLM-REVIEW] Starting Batch Evaluation`) expected by the progress scraper. - **Resolution**: Refactored the core execution logic natively into `src/lib/queue.ts`. ## 🟢 [2026-03-16] Windows 인코딩 이슈 해결 (UTF-8 스트림 및 프론트엔드 mojibake)
- **상황**: Windows Server 패치 리뷰 과정에서 한글이 ????? 또는 깨진 글자(mojibake)로 출력되는 현상 발생.
- **원원**:
  1. queue.ts에서 spawn으로 실행된 자식 프로세스의 stdout, stderr 스트림 인코딩이 명시되지 않아 기본값(Buffer/ASCII)으로 처리됨.
  2. i18n.ts 및 ClientPage.tsx 내의 하드코딩된 한글 문자열이 파일 저장 시 인코딩 오류로 깨짐.
  3. Python 스크립트(windows_preprocessing.py)는 UTF-8로 잘 쓰고 있었으나, 이를 읽어들이는 Node.js 환경에서 인코딩 불일치 발생 가능성.
- **해결 방안**:
  1. queue.ts 내의 모든 spawn 호출부(un*Stream 함수들)에 p.stdout.setEncoding('utf8') 및 p.stderr.setEncoding('utf8')를 명시적으로 추가.
  2. 깨진 i18n.ts 번역문을 정상 한글로 복구하고 파일 인코딩을 UTF-8로 고정.
  3. ClientPage.tsx 내의 깨진 하드코딩 문자열을 i18n 키 또는 정상 한글로 교체.
- **교훈**: **Node.js에서 외부 프로세스(Python 등)와 파이프(spawn)로 통신할 때, 특히 한글 등 멀티바이트 문자가 포함된 데이터를 다룬다면 반드시 스트림 인코딩을 utf8로 명시해야 한다.** 또한, 프론트엔드 파일 내에 직접 한글을 쓸 경우 VS Code/에디터의 파일 인코딩이 UTF-8(BOM 없음)인지 항상 확인하고, 가급적 모든 문자열을 i18n으로 관리하여 인코딩 이슈의 전파 범위를 최소화해야 한다.
